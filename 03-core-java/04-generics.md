# Generics

> Runnable example: [code/core/GenericsDemo.java](../code/core/GenericsDemo.java)

---

## The Big Picture

> **In plain terms** — Generics let you write code that works with *any* type while still being type-safe. Instead of a `Box` that holds a vague `Object` (forcing you to cast and risk crashes), you write `Box<T>` and at use-time say `Box<String>` — now the compiler *guarantees* only strings go in and what comes out is a string, no casts, no surprises. The `<T>` is just a placeholder you fill in later.

> **Why this matters** — Generics move a whole category of bugs from runtime (`ClassCastException` in production) to compile time (red squiggle in your editor) — for free, with zero runtime cost. They're the reason `List<String>` exists and the foundation of the entire Collections framework, streams, and `Optional`. The two ideas that confuse people — *wildcards* (`? extends`/`? super`) and *type erasure* (generics vanish at runtime) — both make sense once you see them as consequences of "type-checking happens at compile time only."

---

## Why Generics Exist

Without generics (pre-Java 5), collections held `Object`. You got runtime errors instead of compile-time errors.

```java
// Without generics:
List list = new ArrayList();
list.add("hello");
list.add(42);
String s = (String) list.get(1); // ClassCastException at runtime — bug found too late

// With generics:
List<String> list = new ArrayList<>();
list.add("hello");
list.add(42); // compile error — caught immediately ✓
String s = list.get(0); // no cast needed
```

Generics give you type safety at **compile time** with **zero runtime overhead** (due to type erasure).

> **In plain terms** — The "before" code compiles fine but blows up when it runs, because the list forgot what it was holding. The "after" code refuses to even compile the mistake. That's the whole pitch: catch the bug while you're typing, not when a user hits it. As a bonus, you stop writing casts everywhere — the compiler already knows the type.

> **Going deeper** — A *raw type* (`List` with no `<...>`) still compiles for backward compatibility with pre-Java-5 code, but it disables generic checking and earns "unchecked" warnings — treat any raw type as a bug to fix. The compiler enforces generics by *inserting the casts for you* behind the scenes, so "no runtime overhead" is literal: the generated bytecode is the same as the hand-cast version, just verified safe. This safety is also why generics integrate so cleanly with [streams](../04-java8-modern/02-streams.md) and lambdas.

## Generic Classes

```java
public class Box<T> { // T is a type parameter — a placeholder for any type
    private T value;

    public Box(T value) { this.value = value; }
    public T get() { return value; }
    public void set(T value) { this.value = value; }

    @Override
    public String toString() { return "Box[" + value + "]"; }
}

// Usage:
Box<String> strBox = new Box<>("hello");
Box<Integer> intBox = new Box<>(42);

strBox.get();    // "hello" — type is String, no cast
intBox.get();    // 42 — type is Integer

// Diamond operator (<>) infers type from context:
Box<List<String>> nested = new Box<>(new ArrayList<>());
```

> **In plain terms** — `<T>` after the class name is a blank you fill in per use: `Box<String>` makes a box of strings, `Box<Integer>` a box of integers — one class, reused safely for every type. The diamond `<>` on the right side just means "compiler, infer the type, I already wrote it on the left."

> **Going deeper** — `T` is a *type parameter*; `String` in `Box<String>` is a *type argument*. Conventions: `T` (type), `E` (element), `K`/`V` (key/value), `R` (result), `N` (number). A type parameter can't be a primitive (`Box<int>` is illegal — use `Box<Integer>`, accepting autoboxing), can't be used in `static` fields (the class is shared across all instantiations), and can't be `new T()` (erasure — see below). Type parameters can also be *recursive* (`<T extends Comparable<T>>`), which looks odd but just means "T can be compared to its own kind."

## Multiple Type Parameters

```java
public class Pair<A, B> {
    private final A first;
    private final B second;

    public Pair(A first, B second) {
        this.first = first;
        this.second = second;
    }
    public A getFirst() { return first; }
    public B getSecond() { return second; }

    @Override
    public String toString() { return "(" + first + ", " + second + ")"; }
}

Pair<String, Integer> nameAge = new Pair<>("Alice", 30);
nameAge.getFirst();   // "Alice" — String
nameAge.getSecond();  // 30 — Integer
```

> **In plain terms** — You can have more than one placeholder. `Map<K, V>` is the everyday example — a key type and a value type. `Pair<A, B>` here holds two independent types, handy for returning two values from a method (recall the [return-multiple-values](../01-basics/04-methods.md#return-types) note).

> **Going deeper** — For a simple immutable data holder like `Pair`, a generic `record` is the modern one-liner: `record Pair<A, B>(A first, B second) {}` — you get the constructor, accessors, `equals`/`hashCode`/`toString` for free. That said, a "pair" with meaningless `first`/`second` names is often a smell — a purpose-named type (`record Coordinate(double lat, double lon)`) reads far better at call sites.

## Generic Methods

The type parameter is declared on the method, not the class.

```java
// Swap two elements in an array
public static <T> void swap(T[] array, int i, int j) {
    T temp = array[i];
    array[i] = array[j];
    array[j] = temp;
}

String[] words = {"hello", "world"};
swap(words, 0, 1); // words is now ["world", "hello"]

// Return a list filled with n copies of an item
public static <T> List<T> repeat(T item, int times) {
    List<T> list = new ArrayList<>();
    for (int i = 0; i < times; i++) list.add(item);
    return list;
}

List<String> greetings = repeat("hi", 3); // ["hi", "hi", "hi"]
```

> **In plain terms** — A method can have its *own* type placeholder, declared in `<T>` just before the return type. This lets a single static helper (like `swap` or `repeat`) work on arrays/lists of any type while keeping everything type-checked. The compiler figures out what `T` is from the arguments you pass.

> **Going deeper** — The `<T>` *before* the return type is what makes it a generic *method* (versus using a class's type parameter) — easy to miss, and the usual reason a "why won't this compile?" generic helper fails. You rarely specify the type explicitly because *inference* handles it, but you can when needed: `Collections.<String>emptyList()`. Generic methods shine for static utilities that shouldn't force the whole class to be generic — most of `java.util.Collections` and `java.util.Arrays` is built this way.

## Bounded Type Parameters

Restrict which types are allowed.

```java
// T must be Number or a subclass (Integer, Double, Long, etc.)
public static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T item : list) {
        total += item.doubleValue(); // can call Number methods because T extends Number
    }
    return total;
}

sum(List.of(1, 2, 3));      // works — Integer extends Number
sum(List.of(1.5, 2.5));     // works — Double extends Number
// sum(List.of("a", "b")); // compile error — String doesn't extend Number

// Multiple bounds: T must extend both Comparable and Serializable
public static <T extends Comparable<T> & Serializable> T max(List<T> list) {
    return Collections.max(list);
}
```

> **In plain terms** — `<T extends Number>` says "T can be any kind of number." The payoff: inside the method you can call `Number` methods (`doubleValue()`) on T, because the compiler now *knows* every T is at least a `Number`. Without the bound, T could be anything and you'd be limited to `Object` methods.

> **Going deeper** — `extends` here means "is-a" for *both* classes and interfaces (there's no separate `implements` in bounds), and you can require several at once with `&` (`<T extends Comparable<T> & Serializable>`). An *unbounded* `<T>` is implicitly `<T extends Object>`, so T always has `toString`/`equals`/etc. Bounds are the mechanism behind sortable generics — `Collections.max` is declared roughly `<T extends Comparable<? super T>>`, which is the "PECS" idea (next) applied to the bound itself so subtypes' comparators work too.

## Wildcards

Wildcards (`?`) represent an unknown type. Used with existing generic types, not when defining them.

```java
// ? extends T — "some subtype of T" — READ-only (producer)
public double sumOfList(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) total += n.doubleValue(); // can read as Number
    return total;
    // list.add(1); // compile error — don't know exact type, can't safely add
}

sumOfList(List.of(1, 2, 3));     // List<Integer> — works
sumOfList(List.of(1.5, 2.5));   // List<Double> — works

// ? super T — "some supertype of T" — WRITE-able (consumer)
public void addNumbers(List<? super Integer> list) {
    list.add(1);   // safe — Integer is a subtype of whatever ? is
    list.add(2);
    // Integer n = list.get(0); // compile error — only Object guaranteed
}

addNumbers(new ArrayList<Integer>());
addNumbers(new ArrayList<Number>());
addNumbers(new ArrayList<Object>());
```

**PECS — Produces Extends, Consumes Super:**
- List is a **producer** (you read from it) → `List<? extends T>`
- List is a **consumer** (you write to it) → `List<? super T>`

> **In plain terms** — Wildcards make your method parameters more flexible about *which* generic type they accept. The mnemonic **PECS** is the whole trick: if the collection *gives you* values (you read), use `? extends T`; if it *receives* values (you write), use `? super T`. A `List<? extends Number>` you can read numbers from but not add to (you don't know the exact subtype); a `List<? super Integer>` you can add integers to but only read back as `Object`.

> **Going deeper** — The reason `List<String>` is *not* a `List<Object>` (generics are *invariant*) is safety: if it were, you could sneak a `42` into a `List<String>`. Wildcards reintroduce controlled flexibility exactly where it's safe. The "can't add to `? extends`" rule trips everyone up — the compiler can't prove your element matches the real (unknown) subtype, so it forbids the write. Use a bounded *type parameter* `<T>` instead of a wildcard when you need to refer to the type more than once (e.g. read one element and add it back); use wildcards for one-directional flexible parameters. The JDK API is full of PECS: `Collections.copy(List<? super T> dest, List<? extends T> src)`.

## Type Erasure

Generics are a compile-time feature only. At runtime, type information is erased.

```java
List<String> strings = new ArrayList<>();
List<Integer> ints = new ArrayList<>();

// At runtime, both are just List — same class
System.out.println(strings.getClass() == ints.getClass()); // true!

// You CANNOT do:
// if (list instanceof List<String>) // compile error — can't check erased type

// You CAN do:
if (list instanceof List<?>) // works — wildcard
```

This also means:
```java
// Can't create generic arrays:
T[] arr = new T[10]; // compile error

// Workaround:
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[10];

// Or better: use List<T> instead of T[]
```

> **In plain terms** — Generics are a *compile-time* feature. Once your code is compiled, the type arguments are thrown away — at runtime a `List<String>` and a `List<Integer>` are both just `List`. This "type erasure" is why you can't ask `list instanceof List<String>` or do `new T[]` — that information simply isn't there anymore when the program runs.

> **Going deeper** — Erasure was chosen for *migration compatibility*: generic and pre-generic code had to interoperate in 2004, so generics had to compile down to the same raw bytecode. The consequences you'll hit: no `T.class`, no `new T()`, no generic arrays, can't `catch` a generic exception type, and two methods that differ only by generic parameter (`foo(List<String>)` vs `foo(List<Integer>)`) clash because they erase to the same signature. The standard workarounds are passing a `Class<T>` token (the "type token" pattern, e.g. `factory(Class<T> type)`) and preferring `List<T>` over `T[]`. Note other languages differ — C#/Kotlin can *reify* generics — so don't carry erasure assumptions across languages.

## Common Generic Patterns in Java APIs

```java
// Comparable<T> — for natural ordering
public class Employee implements Comparable<Employee> {
    @Override
    public int compareTo(Employee other) {
        return this.name.compareTo(other.name);
    }
}

// Iterable<T> — makes your class work in for-each
public class Range implements Iterable<Integer> {
    private final int start, end;
    public Range(int start, int end) { this.start = start; this.end = end; }

    @Override
    public Iterator<Integer> iterator() {
        return new Iterator<>() {
            int current = start;
            public boolean hasNext() { return current < end; }
            public Integer next() { return current++; }
        };
    }
}

for (int n : new Range(1, 5)) {
    System.out.println(n); // 1 2 3 4
}
```

> **In plain terms** — You don't just *consume* generics — you implement the JDK's generic interfaces to plug your own types into the language. Implement `Comparable<T>` and your objects become sortable; implement `Iterable<T>` and your class works in a `for-each` loop. You're filling in the same `<T>` blanks, just on Java's interfaces instead of your own.

> **Going deeper** — This is the generics payoff in practice: by speaking the JDK's generic contracts, your types get first-class language support. `Comparable<T>` powers `Collections.sort` and `TreeSet`; `Iterable<T>` powers for-each and feeds [streams](../04-java8-modern/02-streams.md) (via `StreamSupport`); `Function<T,R>`/`Predicate<T>` (from [lambdas](../04-java8-modern/01-lambdas-and-functional.md)) are all generic functional interfaces. The anonymous `Iterator<>` here uses the diamond on an anonymous class (Java 9+). Designing your *own* generic APIs well — bounded params, PECS wildcards, sensible type-parameter names — is what makes a library pleasant to use.
